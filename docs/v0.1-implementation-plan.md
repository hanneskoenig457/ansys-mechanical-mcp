## Reached Lifecycle and Selection Slice

1. One lazy `MechanicalSessionManager` is carried by the stable FastMCP v1
   lifespan for the stdio server run; Mechanical operations are serialized.
2. Start/connect, version/executable, host, port, batch/UI, transport,
   certificates, cleanup, and ownership are explicit or structurally derived.
   No Mechanical session mode is chosen implicitly. Started headless sessions
   default to owned cleanup; started UI and connected sessions default to
   remaining open.
3. Local `auto` transport resolves and preflights one exact executable before
   any start call. It preserves secure transport on compatible installations,
   stops on unknown evidence, and returns a zero-launch explicit-opt-in error
   for a confirmed legacy SP. One persisted local `insecure` choice is required
   for that case. Connect never auto-downgrades; insecure remote requires a
   separate explicit acknowledgement.
4. A start failure is latched so repeated inspection cannot create a second
   process. Successful session reuse is unchanged; connect-only failures can
   retry.
5. Cleanup is idempotent after success, retryable after failure, and uses the
   documented `Mechanical.exit(force=True)` only when immutable policy permits it.
6. Read-only model inspection and current-selection capture reuse the same
   context. Capture never starts a new empty instance and rejects unknown or
   headless GUI capability.
7. Arbitrary script execution remains internal. No modeling, preprocessing,
   semantic explanation, or mutation was added.
8. Fake-based tests and default-skipped real integration tests cover the new
   boundary honestly.

Stable MCP v1 HTTP lifespans do not have the same process-wide scope, so this
persistent Mechanical lifecycle is intentionally stdio-only for now.

## Exact Next Validation Step

The first explicit-insecure Windows run against Mechanical 2025 R1 SP03
(`R251RC2P03`) has now confirmed one GUI, one launch attempt, and one successful
inspection. It also disproved the assumed loopback-only listener: Windows
reported port 10000 bound to `::` even though the client target was
`127.0.0.1`. The validation correctly stopped before the second inspection and
all selection cases.

The next Windows validation must display the exact listener and owning process
immediately after the one start. Stopping on a non-loopback address remains the
recommended default. The second inspect, session-reuse observation, and
read-only empty, single, multiple, node, element, element-face, and active-tree
selection cases may continue only if the operator explicitly accepts the shown
risk for that one harmless test session on a trusted or isolated computer.
Close Mechanical normally afterward and verify that process and listener are
gone. No Mac fake proves these runtime facts. A later compatible-SP/261 live run
must also confirm that auto retains WNUA rather than weakening transport.

Correct only evidence-backed adapter assumptions before considering target
resolution or any mutation. The first physical action remains intentionally
undecided.

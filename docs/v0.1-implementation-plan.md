# v0.1 Implementation Status and Remaining Plan

## Smallest Useful Prototype

The v0.1 prototype should prove one narrow Static Structural workflow on an
existing Mechanical project. It must not expose broad Ansys APIs or return
simulated solver data.

The public server registers `check_environment`, `inspect_mechanical_model`, and
`capture_current_selection`. Fake-injected unit tests and the in-process MCP
round trip prove the Python/MCP boundaries, not a licensed Mechanical workflow.
The two real Mechanical integration tests are opt-in and skipped by default;
they have not been run successfully in this environment because PyMechanical,
Mechanical, and a license are unavailable.

## Reached Lifecycle and Selection Slice

1. One lazy `MechanicalSessionManager` is carried by the stable FastMCP v1
   lifespan for the stdio server run; Mechanical operations are serialized.
2. Start/connect, version, host, port, batch/UI, cleanup, and ownership are
   explicit. No Mechanical mode is chosen implicitly. Started headless sessions
   default to owned cleanup; started UI and connected sessions default to
   remaining open.
3. Cleanup is idempotent after success, retryable after failure, and uses the
   documented `Mechanical.exit(force=True)` only when immutable policy permits it.
4. Read-only model inspection and current-selection capture reuse the same
   context. Capture never starts a new empty instance and rejects unknown or
   headless GUI capability.
5. Arbitrary script execution remains internal. No modeling, preprocessing,
   semantic explanation, or mutation was added.
6. Fake-based tests and default-skipped real integration tests cover the new
   boundary honestly.

Stable MCP v1 HTTP lifespans do not have the same process-wide scope, so this
persistent Mechanical lifecycle is intentionally stdio-only for now.

# v0.1 Implementation Plan

## Smallest Useful Prototype

The v0.1 prototype should prove one narrow Static Structural workflow on an
existing Mechanical project. It should not expose broad Ansys APIs or return
simulated solver data.

## Tool Sequence

1. `check_environment`
   - Report Python version, platform, relevant installed Python packages, and
     whether Mechanical command-line launch helpers are discoverable.
   - Must run without Ansys installed.

2. `mechanical_start_or_connect`
   - Start Mechanical through PyMechanical or connect to an already running
     Mechanical server.
   - Return product/session metadata only after a real connection succeeds.

3. `mechanical_run_script`
   - Execute a controlled script block or script file through PyMechanical.
   - Treat script execution as a fallback path, not the main API surface.

4. `mechanical_inspect_model`
   - Return basic model and analysis metadata from Mechanical `DataModel`.
   - Start with analysis names, IDs, types, and high-level tree object counts.

5. `static_structural_solve`
   - Locate an existing Static Structural analysis and call Mechanical solve.
   - Return structured status and solver errors; never fake solve completion.

6. `dpf_result_summary`
   - Load a real result file with PyDPF and return analysis metadata, mesh counts,
     available result names, and basic min/max summaries for selected results.

## First Implemented Slice

Implement only `check_environment` first. This slice is intentionally independent
from licensed Ansys products and can be covered by unit tests with injected package
and executable discovery functions.

Implemented surface:

- `ansys-mechanical-mcp --check-environment` prints the structured JSON payload.
- Running `ansys-mechanical-mcp` starts a minimal MCP server over stdio.
- The MCP server currently registers only the `check_environment` tool.

Do not add the Mechanical, Static Structural, or DPF tools until each tool can call
the real official API path described in `docs/api-research.md`.
